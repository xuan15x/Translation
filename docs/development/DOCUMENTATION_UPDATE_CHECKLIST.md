# 文档更新检查清单

**目的**: 确保每次代码修改都同步更新文档，保持文档与代码的一致性  
**适用范围**: 所有代码变更（功能新增、Bug 修复、重构优化等）  
**责任人**: 每次提交的作者

---

## 📋 更新检查流程

### 第一步：识别变更类型

在开始修改前，先确定变更类型：

- [ ] **🚀 新增功能** - 需要更新用户指南、API 文档
- [ ] **✨ 改进优化** - 需要更新相关说明文档
- [ ] **🐛 Bug 修复** - 需要更新故障排查文档
- [ ] **⚠️ 破坏性变更** - 需要更新迁移指南、版本说明
- [ ] **♻️ 代码重构** - 需要更新架构文档、开发指南
- [ ] **🔧 配置变更** - 需要更新配置指南、示例文件
- [ ] **✅ 测试添加** - 需要更新测试文档
- [ ] **📚 文档修正** - 仅更新文档

---

### 第二步：检查并更新版本号

#### 📋 版本号一致性检查（推送前必须执行）

**⚠️ 重要**: 每次推送前必须确保文档版本与代码版本一致！

**检查步骤**:

1. **查看当前代码版本**
   ```bash
   # 查看最新的 Git tag 或提交
   git log --oneline -5
   
   # 或者查看 CHANGELOG 中的最新版本
   grep "^## \[" CHANGELOG.md | head -1
   ```

2. **验证以下文件的版本号**:
   
   | 文件 | 检查位置 | 当前版本 |
   |------|----------|----------|
   | `docs/VERSION.md` | 第 3 行 | v2.2.0 |
   | `CHANGELOG.md` | 第 4 行 | v2.2.0 |
   | `README.md` | 如有版本引用 | - |
   | 各模块文档头部 | 版本标识区 | - |

3. **版本号更新规则**
   ```
   v主版本。次版本.修订版
   
   变更类型 → 版本号更新位置:
   - 新增功能 → 次版本 +1 (v2.2.0 → v2.3.0)
   - Bug 修复 → 修订版 +1 (v2.2.0 → v2.2.1)
   - 破坏性变更 → 主版本 +1 (v2.2.0 → v3.0.0)
   - 文档修正 → 修订版 +1 (v2.2.0 → v2.2.1)
   ```

4. **如果发现版本不一致**:
   ```bash
   # ❌ 错误示例：代码已更新，文档仍为旧版本
   代码提交：v2.3.0 的功能
   文档版本：v2.2.0
   
   # ✅ 正确做法：同步更新文档版本
   编辑 docs/VERSION.md 和 CHANGELOG.md
   将版本号提升至 v2.3.0
   ```

#### 需要更新版本的文件：

- [ ] `docs/VERSION.md` - 文档版本信息
- [ ] `CHANGELOG.md` - 项目更新日志
- [ ] `README.md` - 项目主文档（如果有版本引用）
- [ ] 各模块文档头部的版本标识

检查清单：

- [ ] 更新日期为当前日期
- [ ] 确认适用范围版本号

### 第六步：验证文档一致性

检查以下内容是否一致：

- [ ] 代码中的注释与文档描述一致
- [ ] API 参数说明与实际实现一致
- [ ] 示例代码可以正常运行
- [ ] 截图/图表与当前 UI 一致
- [ ] 引用的文件路径存在且正确
- [ ] 跨文档引用链接有效

---

### 第七步：提交信息规范

使用规范的 Git 提交信息：

```bash
# 格式
<类型>(<范围>): <主题>

[可选的主体内容]

[可选的脚注]

# 示例
docs(config): 更新 API 密钥配置方式至 v2.2.0

- 取消环境变量读取 API 密钥
- 新增配置文件和 GUI 界面配置方式
- 更新相关文档 5 篇

受影响文档:
- docs/guides/CONFIGURATION_CAUTIONS.md
- docs/guides/QUICKSTART.md
- CHANGELOG.md
- docs/VERSION.md

BREAKING CHANGE: 不再支持通过环境变量配置 API 密钥
```

常用类型：
- `feat`: 新功能
- `fix`: Bug 修复
- `docs`: 文档更新
- `style`: 格式调整
- `refactor`: 重构
- `test`: 测试相关
- `chore`: 构建/工具/配置

## 📌 常见错误示例

### ❌ 错误做法

1. **只更新代码，不更新文档**
   ```
   ✗ 代码已修改，文档以后再说
   ```

2. **文档版本不更新**
   ```
   ✗ 代码 v2.2.0，文档仍显示 v2.1.0
   ```

3. **文档内容过时**
   ```
   ✗ 文档中的示例代码无法运行
   ✗ 文档描述的 API 与实际不符
   ```

4. **提交信息不规范**
   ```
   ✗ "update docs"
   ✗ "fix stuff"
   ```

### ✅ 正确做法

1. **代码文档同步更新**
   ```
   ✓ 代码修改完成的同时，文档也已更新
   ✓ 文档版本号已提升至匹配版本
   ```

2. **完整的版本历史**
   ```
   ✓ CHANGELOG.md 记录了所有变更
   ✓ docs/VERSION.md 有详细版本信息
   ```

3. **文档可验证**
   ```
   ✓ 所有示例代码都可以运行
   ✓ 所有截图都是最新的
   ✓ 所有链接都有效
   ```

4. **规范的提交信息**
   ```
   ✓ docs(config): 更新配置文档至 v2.2.0
   ✓ 包含变更详细说明
   ```

## 🔧 附录：推送前版本号检查清单

### ⚠️ 强制检查项（推送前必须完成）

**每次准备推送到 GitHub 前，请按以下顺序检查**:

#### 步骤 1: 查看当前代码版本

```bash
# 查看最近的提交
git log --oneline -5

# 或查看 CHANGELOG 中的最新版本
grep "^## \[" CHANGELOG.md | head -1
```

#### 步骤 2: 验证文档版本号

检查以下文件的版本号是否一致：

| 文件 | 位置 | 要求 |
|------|------|------|
| `docs/VERSION.md` | 第 3 行 | **当前文档版本**: vX.X.X |
| `CHANGELOG.md` | 第 4 行 | **当前版本**: vX.X.X |
| 各文档头部 | 版本标识区 | **文档版本**: vX.X.X |

#### 步骤 3: 如果发现版本不一致

**❌ 禁止推送的情况**:
- 代码已修改为新功能，但文档仍为旧版本
- CHANGELOG 已更新，但 VERSION.md 未更新
- 不同文档的版本号不一致

**✅ 正确的做法**:
1. 先更新所有文档至最新版本
2. 提交文档更新
3. 然后再推送

#### 步骤 4: 快速检查命令

```bash
# Windows PowerShell
Write-Host "Checking versions..."
Select-String -Path "docs\VERSION.md" -Pattern "当前文档版本.*v\d+\.\d+\.\d+"
Select-String -Path "CHANGELOG.md" -Pattern "当前版本.*v\d+\.\d+\.\d+"
```
